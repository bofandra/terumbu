# End-to-End Test Case - Cross-Role Smooth Flow

## Objective

Memastikan alur Terumbu.eco berjalan mulus dari sisi publik sampai workspace internal, dengan perpindahan antar user role yang jelas, data yang tersimpan konsisten, dan akses yang tidak bocor antar role.

Dokumen ini dapat dipakai untuk QA manual, UAT, atau dasar automated Playwright test.

## Primary User Roles

| Role | Demo email | Start URL | Purpose |
|---|---|---|---|
| Guest | none | `/` | Public discovery, campaign browsing, expedition browsing, Academy browsing, guest checkout sanity check. |
| User | `user.demo@terumbu.eco` | `/dashboard` | Donor, expedition participant, Academy learner, Impact Passport owner. |
| Corporate Admin | `corporate.demo@terumbu.eco` | `/corporate/dashboard` | Corporate ESG/CSR workspace, report export, evidence and project monitoring. |
| Partner | `partner.demo@terumbu.eco` | `/partner` | Campaign updates and field evidence submission. |
| Admin | `admin.demo@terumbu.eco` | `/admin` | Platform operation, campaign status, evidence verification, donation reconciliation, audit review. |

Password for role accounts is supplied by the deployment owner and seeded with:

```bash
DEMO_ROLE_PASSWORD="replace-with-shared-password" npm run db:seed-role-users
```

## Test Environment Preconditions

1. Database schema is applied.

```bash
npm run db:push
```

2. Base demo data is seeded.

```bash
npm run db:seed
```

3. Role demo accounts are seeded.

```bash
DEMO_ROLE_PASSWORD="replace-with-shared-password" npm run db:seed-role-users
```

4. App is running.

```bash
npm run dev
```

5. Use a separate browser context per role to avoid session cookie bleed.
6. Use unique test labels for generated records, for example `E2E 2026-07-01 1530 Partner Evidence`.

## Global Pass Criteria

- Every tested page returns a usable rendered page, not a blank page, 404, 500, or unhandled error.
- Protected pages redirect unauthenticated users to `/login?next=...`.
- Login redirects back to the requested `next` path.
- A role only sees or mutates workflows intended for that role.
- Mutations survive page refresh and are visible in the next relevant role workflow.
- Public pages never expose private dashboard data, private Impact Passport content, session-only user data, or admin-only controls.
- Success, error, failed-payment, empty, loading, and forbidden states are understandable and do not trap the user.
- Navigation remains coherent on desktop and mobile widths.

## Access Matrix

| Path group | Guest | User | Corporate Admin | Partner | Admin |
|---|---:|---:|---:|---:|---:|
| `/`, `/campaigns`, `/expeditions`, `/academy`, `/impact-map` | Allow | Allow | Allow | Allow | Allow |
| `/checkout/donation`, `/checkout/expedition` | Allow | Allow | Allow | Allow | Allow |
| `/dashboard/*` | Redirect to login | Allow | Allow | Allow | Allow |
| `/dashboard/passport` | Redirect to login | Allow own passport | Allow own passport | Allow own passport | Allow own passport |
| `/corporate/*` | Redirect to login | No corporate program | Allow | No corporate program | No corporate program unless granted |
| `/partner` | Redirect to login | Forbidden | Forbidden | Allow | Allow |
| `/admin/*` | Redirect to login | Forbidden | Forbidden | Forbidden | Allow |

Expected forbidden behavior: non-admin users attempting `/admin/*` or non-partner users attempting `/partner` are redirected to `/dashboard?error=forbidden`.

## E2E-01 - Public Discovery to User Donation to Dashboard

### Goal

Validate the public campaign journey through checkout, payment recording, user dashboard impact, and receipt/email side effects.

### Steps

1. As Guest, open `/`.
2. Navigate to `/campaigns`.
3. Open any published campaign detail page.
4. Confirm campaign trust content is visible: partner, progress, impact target, updates, evidence, impact site, and donation CTA.
5. Open donation checkout with a coral sponsorship intent, for example `/checkout/donation?campaign={campaignSlug}&intent=coral`.
6. Log in as `user.demo@terumbu.eco` in a fresh user context.
7. Repeat checkout while signed in.
8. Fill name, email, amount, message, and set demo payment result to `Paid`.
9. Submit checkout.
10. Confirm `/checkout/success?status=paid&type=donation...` loads.
11. Open `/dashboard`.
12. Open `/dashboard/donations`.
13. Open `/dashboard/corals`.
14. Open `/dashboard/passport`.
15. Refresh all dashboard pages.

### Expected Results

- Checkout creates a donation linked to the signed-in user.
- Paid donation updates campaign raised amount and donor count.
- Donation receipt is generated.
- Donation receipt email log is queued.
- Coral sponsorship intent creates a sponsored ecosystem record.
- User dashboard shows updated donation/coral impact without requiring logout/login.
- Impact Passport reflects the user's public/private visibility rules.
- Refresh does not lose the newly created records.

## E2E-02 - Failed Donation to Admin Reconciliation

### Goal

Validate failed payment handling, admin reconciliation, receipt generation, campaign totals, and audit log.

### Steps

1. As User, open `/checkout/donation`.
2. Select a campaign and set demo payment result to `Failed`.
3. Submit checkout.
4. Confirm the success page shows failed-payment state.
5. Log out or switch to a clean Admin browser context.
6. Log in as `admin.demo@terumbu.eco`.
7. Open `/admin`.
8. Find the failed donation in donation reconciliation.
9. Change status to `Paid`.
10. Submit `Reconcile`.
11. Open `/admin/audit`.
12. Reopen the public campaign detail page.

### Expected Results

- Failed donation does not immediately increase campaign raised amount.
- Admin can reconcile the donation to paid.
- Campaign totals update after reconciliation.
- Receipt and email log are created after reconciliation.
- Audit log includes `donation.reconciled`.
- User-facing campaign page remains readable after operational changes.

### Expected Results

- Expedition participants are stored.
- Departure `seatsBooked` increases by participant count.

- User dashboard shows the booking.
- Expedition public availability remains consistent after refresh.

## E2E-04 - Academy Enrollment to Certificate to Impact Passport

### Goal

Validate learning progression from enrollment to certificate and passport milestone.

### Steps

1. As User, open `/academy`.
2. Search or filter courses.
3. Open a course detail page.
4. Click `Login to Enroll` if the session is missing, or enroll directly if already signed in.
5. Complete every available lesson.
6. Submit final assessment with a passing score.
7. Open `/dashboard/academy`.
8. Open `/dashboard/certificates`.
9. Open `/dashboard/passport`.
10. Open the public passport URL only if visibility is `public` or `link`.

### Expected Results

- Enrollment is created once and remains idempotent on repeat enrollment.
- Lesson progress reaches 100%.
- Passing assessment creates a certificate.
- Course enrollment becomes completed.
- Certificate appears in dashboard certificates.
- Impact Passport receives a certificate milestone when a passport exists.
- Private passport visibility prevents public discovery.

## E2E-05 - Partner Field Update to Admin Verification to Public Campaign

### Goal

Validate partner submission, admin moderation, audit logging, and public campaign continuity.

### Steps

1. As Partner, open `/partner`.
2. Confirm campaign list, evidence status, and recent updates are visible.
3. Publish a campaign update with a unique title, body, and optional image URL.
4. Submit evidence with a unique title, type, and URL.
5. Confirm the partner portal returns with a saved state.
6. Confirm the new evidence appears with `submitted` status.
7. Attempt to open `/admin`.
8. Switch to Admin context.
9. Open `/admin/evidence`.
10. Find the submitted evidence.
11. Set status to `Verified`.
12. Save.
13. Open `/admin/audit`.
14. Open the related public campaign detail page.
15. Open the public campaign update detail page if the update has a detail link.

### Expected Results

- Partner can publish updates and submit evidence.
- Partner cannot access admin pages.
- Admin can verify or reject submitted evidence.
- Evidence verification writes `evidence.verification.updated` audit log.
- Public campaign detail remains coherent and shows appropriate update/evidence information.
- Verification state does not expose admin notes or internal-only controls publicly.

## E2E-07 - Admin Operations and Guardrails

### Goal

Validate admin-only workflows and ensure operational changes do not break public or user journeys.

### Steps

1. As Admin, open `/admin`.
2. Visit `/admin/campaigns`, `/admin/expeditions`, `/admin/partners`, `/admin/impact-sites`, `/admin/evidence`, `/admin/reports`, `/admin/users`, and `/admin/audit`.

4. Change an organization verification level on `/admin/partners`.
5. Verify or reject a submitted evidence record.
6. Reconcile a donation.
7. Open `/partner`.
8. Open `/dashboard`.
9. Open `/corporate/dashboard`.
10. Open the public homepage and one campaign page.

### Expected Results

- All admin subsections load for Admin.
- Admin actions create or preserve audit-log records.
- Admin can inspect partner portal because `/partner` allows `admin`.
- Admin can access own dashboard.
- Corporate dashboard shows configured corporate data only if Admin has corporate permission; otherwise it shows a no-program state.
- Public pages remain stable after admin operational changes.

## E2E-08 - Session, Redirect, and Logout Smoothness

### Goal

Validate authentication ergonomics and role separation.

### Steps

1. As Guest, open `/dashboard/passport`.
2. Confirm redirect to `/login?next=%2Fdashboard%2Fpassport`.
3. Log in as User.
4. Confirm redirect lands on `/dashboard/passport`.
5. Open `/login` while already signed in.
6. Confirm already-signed-in behavior redirects to the safe next path.
7. Log out.
8. Confirm `/login?loggedOut=1` appears.
9. Attempt browser back to a protected route.
10. Repeat with Partner attempting `/admin/evidence`.
11. Repeat with Corporate Admin attempting `/partner`.

### Expected Results

- `next` redirect is preserved for valid internal paths.
- Invalid or external next paths fall back to `/dashboard`.
- Logout destroys the session cookie.
- Browser back does not reveal protected rendered content.
- Forbidden redirects are consistent and readable.

## E2E-09 - Public Impact Passport Privacy

### Goal

Validate Impact Passport visibility modes across private, link, and public settings.

### Steps

1. As User, open `/dashboard/settings`.
2. Set passport visibility to `Private`.
3. Open `/dashboard/passport` and copy the current public slug if visible in the UI.
4. Open `/passport/{publicSlug}` from a Guest context.
5. Return as User and set visibility to `Link`.
6. Open `/passport/{publicSlug}` from a Guest context.
7. Return as User and set visibility to `Public`.
8. Open `/passport/{publicSlug}` from a Guest context.
9. Open homepage or passport preview surfaces if available.

### Expected Results

- Private passport is not publicly readable.
- Link visibility allows direct URL access but should not be promoted as a public discovery item.
- Public visibility allows public profile rendering.
- Dashboard remains the only place where private account controls appear.
- Visibility changes survive refresh and logout/login.

## Defect Severity Guide

| Severity | Definition | Examples |
|---|---|---|
| P0 | Blocks core business flow or exposes private data. | User cannot donate; admin pages exposed to user; private passport public. |
| P1 | Major role workflow broken. | Partner cannot submit evidence; admin cannot verify; corporate export cannot be queued. |
| P2 | Flow works but is confusing or inconsistent. | Saved state missing, stale dashboard after refresh, unclear failed-payment copy. |
| P3 | Visual polish or low-risk content issue. | Minor spacing issue, non-critical copy inconsistency, optional image missing. |

## Exit Criteria

The end-to-end test pass is accepted when:

1. All P0 and P1 issues are fixed or explicitly accepted by product owner.
2. All cross-role workflows above pass in a fresh seeded environment.
3. Every mutation has a visible confirmation and a verifiable downstream effect.
4. Access matrix behavior is confirmed for every demo role.
5. Public pages remain stable after admin, partner, corporate, and user mutations.
